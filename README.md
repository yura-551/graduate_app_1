# graduate_app_1

## 1. サービス概要

- いくつかの質問に答えると、その人の体調・気分に合ったアロマを提案してくれるアプリ。
- 「なんとなく不調・疲れているけど、どのアロマを選べばいいか分からない」という人の、アロマ選びの迷いを解決する。

- 誰の / どんな課題を / どう解決するか
  疲れやストレスをアロマでケアしたい人の「種類が多すぎて、今の自分に合うものが選べない」という課題を、質問に答えるだけで体調・気分に合ったアロマを提案するアプリで解決する。

## 2. このアイデアはどこから生まれたか
### 2-1. きっかけとなった体験・感情

このアイデアの元になった
自分自身の体験・感情・違和感を教えてください。

仕事でアロマオイルを使って患者様のマッサージを行うことがあり、先輩と一緒に、その時の患者様の気分・体調に合わせてアロマを調合した経験がある。実際に気分に合わせて選んだアロマで患者様が楽になる様子を見て効果を実感した一方で、「どのアロマを選ぶか」の判断には知識と経験が必要で、その場で適切に選ぶのは簡単ではないと感じた。もっと手軽に、その人の気分・体調に合ったアロマ選びができれば、選ぶ側も使う側も楽になるのではないかと考えた。

### 2-2. なぜそれが「気になった」のか
同じ体験をしても、気にしない人もいます。
なぜ自分はそれが気になったのだと思いますか？

- 現場でアロマの効果を実感したからこそ、「気分に合わせて選ぶ」という価値が分かった。同時に、その選択には知識と経験が必要で、詳しくない人には難しい・遠ざかってしまう領域だと感じた。効果を知っているからこそ、この選ぶハードルをもっと下げたいと思ったから気になった。


## 3. 課題の整理（表面的になっていないか）
### 3-1. 表に見えている困りごと

その体験の中で、最初に感じた「困りごと」は何ですか？
アロマの種類が多すぎて、今の自分の体調・気分に合うものが選べなかった。

### 3-2. 本当に解決したい課題は何か
「効果があるのは分かっているのに、選び方が分からず結局使わなくなる」ことを改善したい。

「なぜそれが困るのか？」を**最低3回**繰り返してください。

- なぜ？　アロマオイルごとの効果（リラックス・安眠・気分転換など）を一つずつ調べるのは手間で、疲れているときほど調べる気力がない。

- なぜ？　調べる気力を失うと、疲れやストレスを我慢するしかなくなり、心身の回復が後回しになってしまう。

→ 本当に解決したい課題：疲れているときでも迷わず、今の自分に合うアロマにたどり着けるようにしたい。

※「アプリでできそうなこと」ではなく
※「人の行動・感情の問題」になっているか確認してください。

### 5-1. 似たサービスの調査

5-2. それでも自分が作りたい理由

❌ 悪い例：「タグ機能がある」「通知機能がある」
⭕ 良い例：「料理初心者が冷蔵庫の残り物だけでレシピを探せる」「忙しい人が寝る前3分で振り返れる」

このアプリは、【どんな人】の【どんな瞬間】を一番助けるアプリですか？
- 私のアプリの差別化ポイント：
  「疲れて帰宅した20〜30代の働く女性」が「アロマで少し楽になりたいと思った瞬間」に、精油の効能を一つずつ調べ回ることなく、いくつかの質問に答えるだけで今の自分に合うアロマを提案してもらえる。既存サービスが「効能を調べる／商品を買う」のに対し、こちらは今の体調・気分から逆算して提案し、ログインで記録を残せる点が違う。

## 7. このアプリで実現すること

まずアプリを１つ完成させる。機能は盛らずに「アカウント登録・ログインして、質問に答えると自分に合ったアロマが提案される」というコア価値だけに絞る。

### MVPで作る機能（今回作りきる範囲）

- アカウント登録・ログイン／ログアウト（メールアドレス＋パスワード）
- 診断（いくつかの質問に答える。例：今の気分／体の状態／求めたい効果 など）
- 診断結果に応じたアロマの提案（1〜3件。名前・効果・簡単な使い方を表示）
- アロマの詳細ページ（効果・使い方・向いている場面）
- 診断履歴の保存・一覧（ログインユーザーが過去の診断結果を振り返れる）

補足：
- アロマのデータは手入力で10〜15件からスタート
- 診断ロジックは複雑にせず、質問への回答→タグ／スコアでの単純なマッチングから始める
- 購入機能・決済・外部連携・SNSシェア・レコメンドの高度化は今回は作らない

### 本リリースで作る機能（余裕があれば足す）

- 診断ロジックの精緻化（複数条件の重み付け、より細かい提案）
- お気に入り保存、使ってみた記録・メモ
- 気分の変化の記録・振り返り（時系列での可視化）
- アロマの種類・データの拡充、ブレンド提案

## 8. このアプリの懸念点とその対策
※最低1つは必ず書いてください。
### 懸念点①：アロマの効果・提案内容の信頼性

- 何が問題になりそうか：アロマの効果には個人差があり、「この状態にはこれが効く」と断定すると誤解やトラブルの元になる。
- なぜそう思うか：精油は体質・体調（妊娠中・持病・アレルギー等）によって注意が必要なものもあり、医療的な効果を保証するものではないため。
- そのために考えている対策：効果は「一般的にこう言われている」という事実ベースの表現にとどめ、断定を避ける。注意書き（医療行為ではない旨・利用上の注意）を明記する。出典のある情報をもとにデータを整える。

### 懸念点②：診断ロジックの妥当性

- 何が問題になりそうか：質問への答えと提案するアロマの対応が的外れだと、「合っていない」と感じて使われなくなる。
- なぜそう思うか：体調・気分は主観的で幅があり、少ない質問数で精度を出すのが難しいため。
- そのために考えている対策：MVPでは質問と効果タグの対応をシンプルに設計し、まず「それっぽく納得できる」提案を優先する。フィードバックをもらいながら質問・対応を改善していく。

## 9. 今後の展開・発展の方向性

このアプリは、
「使って終わり」にならないために
どんな広がりが考えられますか？

### 観点（すべて答えなくてOK）

- ユーザーはなぜ使い続ける？
- 続けることで何が変わる？
- データが溜まると何ができる？
- 他の人と関わる余地はある？
- 運用・仕組みで価値を出せないか？

- ユーザーはなぜ使い続ける？→体調・気分は日々変わるため、その日ごとに診断でき、繰り返し使う理由がある。
- データが溜まると何ができる？→診断履歴が溜まれば「自分に合いやすい傾向」が見え、提案の精度を上げられる。気分の変化の振り返りにも使える。
- 他の人と関わる余地は？→「使ってよかったアロマ」の共有や口コミが集まれば、提案の裏付けが厚くなる。

### 今考えている発展案

- 機能追加の方向性：診断ロジックの高度化、お気に入り・使用記録、気分の時系列記録、ブレンド提案、通知でのセルフケア習慣化。
- UI・体験の改善：診断フローの分かりやすさ、結果画面の見やすさ、履歴の振り返りやすさ。
- 機能以外の工夫：正確なアロマデータの整備と出典明記による信頼性の担保、利用上の注意の明示。

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

- フレームワーク：Ruby on Rails（登録・ログイン、診断、一覧・詳細ページ）
- 認証：devise（アカウント登録・ログイン機能を短期間で実装できる定番ライブラリ）
- DB：SQLite（MVPは件数が少ないので手軽なSQLiteで十分。本番はPostgreSQLも検討）
- デプロイ先：Render（無料枠から始められる）
- 使用予定ライブラリ：devise（認証）、（必要に応じて）simple_form などフォーム補助

※アロマデータは10〜15件と少ないため、最初は本格的なDB設計をせず、seed（初期データ）にまとめて持つところから始める。診断ロジックも「回答→タグ／スコアでのマッチング」という単純な形から始め、まず「動くもの」を優先する。

### 10-2. 技術選定の比較検討

使用する技術を列挙するだけでなく、なぜその技術を選んだのかを比較して書いてください。

- 他にどんな技術を候補にしたか
- なぜその候補ではなく、この技術を選んだのか
- 不採用にした技術とその理由

| 候補 | 採用/不採用 | 理由 |
| --- | --- | --- |
| Ruby on Rails | 採用 | 学習リソースが豊富で、ログイン・診断・一覧/詳細（CRUD）を短期間で作れる。今回の規模に合っている |
| Django (Python) | 不採用 | 十分作れるが、学習の連続性・教材の多さでRailsを優先 |
| React / Next.js | 不採用（今回は見送り） | 診断UIの体験改善には有効だが、MVPで導入するとキャッチアップに時間を取られる。「まず1つ作りきる」目標を優先（10-3参照） |
| devise（認証） | 採用 | ログイン機能を安全かつ短期間で実装できる定番ライブラリ。自前実装よりリスクが低い |
| 認証を自前実装 | 不採用 | パスワード管理などセキュリティのリスクが高く、学習コストも大きい。今回の目的に対して割に合わない |
| SQLite | 採用 | 件数が少なく手軽に始められる。MVPの規模には十分 |
| PostgreSQL | 採用候補（本番） | Renderでの本番運用や将来のデータ増加を見据えると有力。MVPはSQLite、本番で切り替えを検討 |

- 今回チャレンジしたい点：deviseでログイン機能を作り、ログインユーザーの診断結果を保存・履歴表示できる形を作りきること。
- 不安な点：診断ロジック（質問→アロマ提案）の設計と、正確なアロマデータを手作業でそろえること。

### 10-3.キャッチアップ不足の懸念

キャッチアップ不足の段階では以下の懸念点があります。
1. 完成までのハードルが高い
使用する言語やフレームワークが異なるため、キャッチアップにかかる時間は個人差があるものの、一定の時間を要することが予想される。
2. 開発期間内に完了することが難しい
「どこまで理解していて、何が分からないのか」「調査にかかる時間」などが不明瞭な場合、開発期間内に完了することが難しくなってしまう。
