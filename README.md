# graduate_app_1

## 1. サービス概要

- 厄除け・厄払いに目的を絞って、対象エリアの神社をマップでみれるアプリ。
- 「厄払いに行きたいけれど、どの神社がいいか調べるのが億劫」という人の、神社探しの手間を解決する。
- Google検索でサイトを1つずつ開いて回る手間をなくし、目的に合った神社をすぐに見つけられるようにする。

- **誰の / どんな課題を / どう解決するか**
  厄払いに行きたい人の「効果がありそうな神社を調べるのが大変」という課題を、厄除けに特化した神社マップで解決する。

## 7. このアプリで実現すること

まずアプリを１つ完成させる。機能は盛らずに「厄除け神社と地図を一覧でみられる」というコア価値だけに絞る

### MVPで作る機能（今回作りきる範囲）

- 厄除け・厄払いに関係する神社の一覧表示（名前・地域・特徴）
- 地図表示（対象エリアの神社をピンで表示。ピンを押すと詳細へ）
- 神社の詳細ページ（住所・アクセス・公式サイトへのリンク）

補足：
- 対象エリアは絞る（例：東京23区）
- データは手入力で10件～15件からスタート
- 検索・現在地からの距離順ソート・比較・口コミ・ログインは今回は作らない

### 本リリースで作る機能（余裕があれば足す）

- 現在地から近い順の並び替え（距離検索）
- 一覧での比較（特徴・アクセスなどを見比べる）
- 口コミ・評価、お気に入り保存
- 対象エリアの拡大

## 8. このアプリの懸念点とその対策
※最低1つは必ず書いてください。
### 懸念点①：神社データの網羅性・正確性

- 何が問題になりそうか：厄除け神社の情報（住所・ご利益・特徴）を、正確かつ十分な数そろえられるか。
- なぜそう思うか：神社の公式情報は形式がバラバラで機械的に集めにくく、間違った情報を載せると信頼を損なうため。
- そのために考えている対策：MVPではエリアを絞る。まず狭く正しく作り、あとから範囲を広げる。

### 懸念点②：「効果がありそう」を何で示すか

- 何が問題になりそうか：厄除けの「効果」は客観的に測れないため、比較の軸が曖昧になりやすい。
- なぜそう思うか：効果を断定するとトラブルの元になり、逆に情報が薄いと比較の意味が薄れるため。
- そのために考えている対策：効果を断定せず、「厄除けのご利益を公式に掲げているか」「アクセスのしやすさ」など事実ベースで比較できる軸に置き換える

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

- フレームワーク：Ruby on Rails（一覧・詳細ページ）
- DB：SQLite（MVPは件数が少ないので手軽なSQLiteで十分）
- デプロイ先：Render（無料枠から始められる）
- 使用予定ライブラリ：Leaflet + OpenStreetMap（無料・登録不要で地図とピンを表示できる）

※神社データは10〜15件と少ないため、最初は本格的なDB設計をせず、seed（初期データ）にまとめて持つところから始める。まず「動くもの」を優先する。

### 10-2. 技術選定の比較検討

使用する技術を列挙するだけでなく、**なぜその技術を選んだのか**を比較して書いてください。

- 他にどんな技術を候補にしたか
- なぜその候補ではなく、この技術を選んだのか
- 不採用にした技術とその理由

| 候補 | 採用/不採用 | 理由 |
| --- | --- | --- |
| Ruby on Rails | 採用 | 学習リソースが豊富で、一覧・詳細ページ（CRUD）を短期間で作れる。今回の規模に合っている |
| Django (Python) | 不採用 | 十分作れるが、学習の連続性・教材の多さでRailsを優先 |
| React / Next.js | 不採用（今回は見送り） | 体験改善に有効だが、MVPで導入するとキャッチアップに時間を取られる。「まず1つ作りきる」目標を優先（10-3参照） |
| SQLite | 採用 | 件数が少なく手軽に始められる。今回は本格的な位置情報検索をしないため十分 |
| PostgreSQL | 不採用（今回は見送り） | 距離検索まで作り込むなら有力だが、今回はそこまでやらないためオーバースペック |
| Leaflet + OpenStreetMap | 採用 | 無料・登録不要で地図とピンを表示でき、学習コストが低い |
| Google Maps API | 不採用 | 情報は豊富だが従量課金・キー登録が必要で、今回の規模には不要 |

- 今回チャレンジしたい点：地図（Leaflet）にピンを立て、押すと神社の詳細が見られる形を作りきること。
- 不安な点：地図ライブラリのキャッチアップと、正確な神社データを手作業でそろえること。

### 10-3.キャッチアップ不足の懸念

キャッチアップ不足の段階では以下の懸念点があります。
1. 完成までのハードルが高い

「どこまで理解していて、何が分からないのか」「調査にかかる時間」などが不明瞭な場合、開発期間内に完了することが難しくなってしまう。
