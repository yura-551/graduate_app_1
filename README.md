サービス名：AromaTune

いくつかの質問に答えると、アーユルヴェーダに基づいて、
その人の体調・気分に合ったアロマを提案してくれるアプリ。

## 1. サービス概要

「なんとなく不調・疲れているけど、どのアロマを選べばいいか分からない」という人の、アロマ選びの迷いをアーユルヴェーダに基づいたアロマ診断で解決する。自分の好みだけでなく体質に合わせたものを提案することで、より効果を実感できるようなアロマを効率的に探せる。質問に答えるだけで体調や気分に合ったアロマを提案するもので、質問や回答での医学的明言は行わない。裏側のデータでは効能で整理するが、ユーザー向けには断定せずに『あなたに合うものはこれでしょう』という提案する。

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

仕事でアロマオイルを使って患者様のマッサージを行うことがあり、先輩と一緒に、その時の患者様の気分・体調に合わせてアロマを調合した経験がある。実際に気分に合わせて選んだアロマで患者様が楽になる様子を見て効果を実感した一方で、「どのアロマを選ぶか」の判断には知識と経験が必要であり。もっと手軽に、その人の気分・体調に合ったアロマ選びができれば、選ぶ側も使う側も楽になるのではないかと考えた。

### 2-2. なぜそれが「気になった」のか

現場でアロマの効果を実感したからこそ、「気分や体調に合わせて選ぶ」という価値が分かった。同時に、その選択には知識と経験が必要で、詳しくない人には難しい・遠ざかってしまう領域だと感じた。この選ぶハードルをもっと下げたいと思ったから気になった。

## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと

アロマの種類が多すぎて、今の自分の体調・気分に合うものが選べなかった。

### 3-2. 本当に解決したい課題は何か

「効果があるのは分かっているのに、選び方が分からず結局使わなくなる」ことを改善したい。

- なぜ？ アロマオイルごとの効果（リラックス・安眠・気分転換など）を一つずつ調べるのは手間で、疲れているときほど調べる気力がない。
- なぜ？ 調べる気力を失うと、疲れやストレスを我慢するしかなくなり、心身の回復が後回しになってしまう。

本当に解決したい課題：疲れているときでも迷わず、今の自分に合うアロマにたどり着けるようにしたい。

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

想定する1人：仕事終わりに疲れを感じている、20〜30代の一人暮らしの働く女性。SNSやドラッグストアでアロマに興味を持ったが、種類が多くて自分用に何を買えばいいか決めきれていない。

年齢設定：

- 年齢：20代〜30代
- 理由：仕事のストレスや疲れを感じやすく、セルフケアへの関心が高い年代。スマホで手軽に完結するサービスに馴染みがあり、アロマも「まず気軽に試したい」というニーズが強いと思うから。

生活・状況設定：

- 生活・状況：仕事で疲れて帰宅し、少しでもリラックスしたい／気分を切り替えたい状況
- 理由：「疲れているけど何を選べばいいか分からない」という場面で、質問に答えるだけで今の自分に合うアロマが分かるため、調べる労力なくケアの一歩を踏み出せる。

### 4-2. 自分とユーザーの距離

自分自身が現場でアロマを使って患者様をケアし、気分・体調に合わせた調合を経験してきた。だからこそ「気分に合わせて選ぶと効果がある」という価値と、「その選択には知識・経験が要る」という難しさの両方を実感として知っている。
その経験をもとに、詳しくない人でも迷わず選べる形にしたいと考えた。

### 4-3. 実際に使われる可能性について（需要・存在確認）

- まず最初に使ってくれそうな人は誰か：アロマに興味はあるが選び方が分からない人／疲れ・ストレスをケアしたい人
- どんな場面でこのアプリを思い出すか：疲れて帰宅したとき、眠れないとき、気分を切り替えたいとき
- 今の生活の中で、代わりに使っているものは何か：Google検索やSNS、店員に相談、なんとなくの直感で選ぶ
- それを置き換えてまで使う理由はあるか：質問に答えるだけで、その時々の体調・気分に合ったアロマが分かり、調べる手間なく選べる。ログインすれば過去の診断も振り返れる。

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ（導入・継続の流れ）

- 知る・思い出すきっかけ：疲れ・ストレス・不眠を感じ、「アロマを試したいけど何を選べばいいか分からない」と思ったとき
- 最初の一歩（登録・導入）：アカウントを作成してログインし、いくつかの質問に答える（診断結果はアカウントに紐づけて保存される）
- 利用を続ける理由：体調・気分は日によって変わるため、その日ごとに診断でき、過去の結果と比べながら自分に合うアロマを見つけていける

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

アロマの通販サイトや精油の効能一覧、性格診断系のコンテンツはある。

- アットアロマ公式アプリ
- 生活の木 公式アプリ
- アロマAI診断

### 5-2. それでも自分が作りたい理由

既存のものは「精油の効能を調べる」「商品を買う」ためのものが多く、今この瞬間の自分の体調・気分から逆算して選ぶ導線が弱い。また、ログインして日々の体調の記録を残せるサービスがあればいいと思った。

- アットアロマアプリ：そのときの気分で選んでいく傾向が強く、体調に関する選択肢がほぼない。記録はできない。
- 生活の木アプリ：主にアロマ関連のワークショップやショッピングのためのアプリ。診断は主に気分で選ぶようになっている。記録はできない。
- アロマAI診断：AIで間違うことがある。オイルの効能・禁忌（妊娠中・肌への希釈など）を自分でデータ化し、AIに頼らず、自分で整備した安全なデータで診断する。

### 5-3. 差別化を「体験価値」で表現する

このアプリは、【どんな人】の【どんな瞬間】を一番助けるアプリか。

「疲れて帰宅した20〜30代の働く女性」が「アロマで少し楽になりたいと思った瞬間」に、精油の効能を一つずつ調べ回ることなく、いくつかの質問に答えるだけで今の自分に合うアロマを提案してもらえる。既存サービスが「効能を調べる／商品を買う」のに対し、こちらは今の体調・気分から逆算して提案し、ログインで記録を残せる点が違う。

## 6. このサービスで提供したい価値

### 6-1. ユーザーの変化

- 行動の変化：種類の多さに迷って諦めていたのが、質問に答えてすぐにアロマを選び、使い始められる。
- 気持ちの変化：「どれを選べばいいか分からない」という迷いが減り、「今の自分にはこれが合う」という納得感・安心感が得られる。
- 考え方の変化：アロマを「詳しい人向けの難しいもの」から「今の自分の状態に合わせて手軽に選べるセルフケア」として捉えられるようになる。

### 6-2. 価値を一文で表す

このサービスは、疲れやストレスをアロマでケアしたい人が、質問に答えるだけで今の自分の体調・気分に合ったアロマを迷わず見つけられるようになるサービス。

## 7. このアプリで実現すること

まずアプリを1つ完成させる。まずは機能は盛らずに「アカウント登録・ログインして、質問に答えると自分に合ったアロマが提案される」というコア価値だけに絞る。

### MVPで作る機能（今回作りきる範囲）

- アカウント登録・ログイン／ログアウト（メールアドレス＋パスワード）
- 診断（いくつかの質問に答える。例：今の気分／体の状態／求めたい効果 など）
- 診断結果に応じたアロマの提案（1〜3件。名前・効果・簡単な使い方を表示）
- 診断履歴の保存・一覧（ログインユーザーが過去の診断結果を振り返れる）

補足：

- アロマのデータは手入力で10〜15件からスタート。
- 診断ロジックは複雑にせず、質問への複数の回答からマッチングして始める。回答ごとにスコア化し、最大値になるアロマを提案する。
- 例：インドのアーユルヴェーダにならい、まずは体質の診断を行う。3〜4回の回答で体質を絞ったら、そのあとはその時の症状や気持ちに対する質問を行い、回答してもらう。（疲れやすい → 手足が冷える → 血行不良に関する回答が多い → 体質に適したかつ血行不良に効能があるアロマを提案する、までが理想）
- 購入機能・決済・外部連携の高度化は今回は作らない。

### 本リリースで作る機能（余裕があれば足す）

- 診断ロジックの精緻化（複数条件の重み付け、より細かい提案）
- 気分の変化の記録・振り返り（時系列での可視化）
- アロマの種類・ブレンド提案

## 8. このアプリの懸念点とその対策

### 懸念点1：アロマの効果・提案内容の信頼性

- 何が問題になりそうか：アロマの効果には個人差があり、「この状態にはこれが効く」と断定すると誤解やトラブルの元になる。そのため、症状に対してこういった効能がありますといった断定はしない。
- なぜそう思うか：アロマオイルは体質・体調（妊娠中・持病・アレルギー等）によって注意が必要なものもあり、医療的な効果を保証するものではないため。
- 対策：効果は「一般的にこう言われている」という事実ベースの表現にとどめ、断定を避ける。注意書き（医療行為ではない旨・利用上の注意）を明記する。出典のある情報をもとにデータを整える。

### 懸念点2：診断ロジックの妥当性

- 何が問題になりそうか：質問への答えと提案するアロマの対応が的外れだと、「合っていない」と感じて使われなくなる。
- なぜそう思うか：体調・気分は主観的で幅があり、少ない質問数で精度を出すのが難しいため。
- 対策：MVPでは質問と効果タグの対応をシンプルに設計し、まず「それっぽく納得できる」提案を優先する。フィードバックをもらいながら質問・対応を改善していく。

## 9. 今後の展開・発展の方向性

### 観点

- ユーザーはなぜ使い続ける：体調・気分は日々変わるため、その日ごとに診断でき、繰り返し使う理由がある。
- データが溜まると何ができる：診断履歴が溜まれば「自分に合いやすい傾向」が見え、提案の精度を上げられる。気分の変化の振り返りにも使える。
- 他の人と関わる余地：「使ってよかったアロマ」の共有や口コミが集まれば、提案の裏付けが厚くなる。

### 今考えている発展案

- 機能追加の方向性：診断ロジックの高度化、お気に入り・使用記録、気分の時系列記録、ブレンド提案、通知でのセルフケア習慣化。
- UI・体験の改善：診断フローの分かりやすさ、結果画面の見やすさ、履歴の振り返りやすさ。
- 機能以外の工夫：正確なアロマデータの整備と出典明記による信頼性の担保、利用上の注意の明示。

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

- フレームワーク：Ruby on Rails　
- DB：postgreSQL
- デプロイ先：Render
- 使用予定ライブラリ：必要に応じて devise(認証)、simple_form などフォーム補助

### 10-2. 技術選定の比較検討

| 候補 | 採用/不採用 | 理由 |
| --- | --- | --- |
| Ruby on Rails | 採用 | 学習リソースが豊富で、ログイン・診断・一覧/詳細（CRUD）を短期間で作れる。今回の規模に合っている |
| Django (Python) | 不採用 | 十分作れるが、学習の連続性・教材の多さでRailsを優先 |
| React / Next.js | 不採用（今回は見送り） | 診断UIの体験改善には有効だが、MVPで導入するとキャッチアップに時間を取られる。「まず1つ作りきる」目標を優先（10-3参照） |
| devise（認証） | 採用 | ログイン機能を安全かつ短期間で実装できる定番ライブラリ。自前実装よりリスクが低い |
| 認証を自前実装 | 不採用 | パスワード管理などセキュリティのリスクが高く、学習コストも大きい。今回の目的に対して割に合わない |
| SQLite | 採用 | 件数が少なく手軽に始められる。MVPの規模には十分 |
| PostgreSQL | 採用候補（本番） | Renderでの本番運用や将来のデータ増加を見据えると有力。MVPはSQLite、本番で切り替えを検討 |

- 今回チャレンジしたい点：deviseでログイン機能を作り、ログインユーザーの診断結果を保存・履歴表示できる形を作りきること。
- 不安な点：診断ロジック（質問 → アロマ提案）の設計と、正確なアロマデータを手作業でそろえること。

### 10-3. キャッチアップ不足の懸念

キャッチアップ不足の段階では以下の懸念点があります。

1. 完成までのハードルが高い。使用する言語やフレームワークが異なるため、キャッチアップにかかる時間は個人差があるものの、一定の時間を要することが予想される。
2. 開発期間内に完了することが難しい。「どこまで理解していて、何が分からないのか」「調査にかかる時間」などが不明瞭な場合、開発期間内に完了することが難しくなってしまう。
